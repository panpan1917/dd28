# dd28 改造迁移总规划

## 1. 文档目的

本文档用于定义 `dd28` 老系统向新架构迁移的总体蓝图、技术路线、模块边界、数据策略、阶段路线和风险控制方案。

本规划采用以下主方案：

- 后端：`Kotlin + Spring Boot`
- 用户端：`Flutter` 同时覆盖 `Web / iOS / Android`
- 架构：`API-first` 的模块化单体
- 基础设施：`MySQL + Redis + RabbitMQ + WebSocket`

结合当前仓库存在独立后台系统这一事实，建议同步建设：

- 后台管理端：`Next.js + TypeScript`

说明：

- `Next.js` 只承担后台运营和需要高效表格交互的 Web 端，不改变主方案。
- 用户主站、登录后业务、移动端体验仍以 `Flutter` 为核心。

## 2. 当前系统现状摘要

### 2.1 当前仓库特征

从当前仓库可见，该系统并不是普通内容站，而是完整的竞猜/钱包/运营平台：

- 入口分流位于 `main/index.php`，按设备跳转到 `pcindex.php` 或 `mobile.php`
- PC MVC 入口为 `main/b.php`
- 移动 MVC 入口为 `main/mobile.php`
- 后台入口为 `admin/index.php`
- 启动核心位于 `core/ini.php`
- 用户登录态与余额刷新逻辑集中在 `lib/mobile/action/BaseAction.php`
- 核心下注与实时轮询逻辑集中在 `lib/mobile/action/GameAction.php`
- 数据快照位于 `kdydata.sql`

### 2.2 已确认的复杂度

- `kdydata.sql` 中可检出的 `CREATE TABLE` 语句数量为 `333`
- `admin/` 根目录下有约 `91` 个文件
- `caiji/` 根目录下有约 `25` 个采集/跑批脚本
- `lib/mobile/action/` 下有 `13` 个移动端业务 Action
- `lib/index/action/` 下有 `6` 个 PC 端业务 Action

### 2.3 当前系统的结构性问题

1. 入口和表现层混杂
   - 旧式直出页、MVC 控制器、后台脚本页并存
   - PC、移动、后台三套体系耦合数据库，但技术实现风格不同

2. 业务规则沉入数据库
   - 登录、提现、下注、开奖、返利等关键流程大量依赖存储过程
   - 应用层缺乏清晰的领域边界与统一事务模型

3. 游戏数据模型高度复制
   - 以 `game28 / gamepk10 / gamecan28` 等为代表，存在大量按游戏拆开的表族
   - 玩法扩展、统计报表、统一风控和统一结算都很困难

4. 实时逻辑分散
   - 开奖采集在 `caiji/`
   - 下注逻辑在 `lib/mobile/action/`
   - 后台运营、人工处理又在 `admin/`
   - 同一个业务流跨 PHP 页面、脚本、SQL 过程和定时任务

5. 安全与可维护性不足
   - 参数过滤主要依赖正则拦截
   - SQL 大量手写拼接
   - 配置和环境隔离能力弱
   - 缺少自动化测试、统一监控和现代发布流程

## 3. 改造目标

### 3.1 业务目标

- 在不打断现有业务运营的前提下完成核心系统迁移
- 建立可持续扩展的新平台，支持后续新增游戏、活动和代理体系
- 将“登录、账户、账务、下注、开奖、返利、后台运营”纳入统一治理

### 3.2 技术目标

- 把核心业务规则从存储过程迁回应用层
- 建立清晰的领域模块和 API 边界
- 将前端、后台、任务调度、实时推送纳入统一技术栈
- 建立可测试、可观测、可灰度发布的新架构

### 3.3 迁移目标

- 老系统在迁移期保持可用
- 新老系统分阶段并行
- 每个阶段都有明确交付物、回滚方案和验收门槛
- 最终完成生产切换，并逐步下线旧 PHP 逻辑

## 4. 设计原则

1. 先保业务，再做重构
   - 不追求一次性重写全部功能
   - 优先迁移高价值、高风险、强一致性链路

2. 模块化单体优先
   - 先做边界清晰、部署简单的模块化单体
   - 不在第一阶段引入微服务拆分成本

3. API First
   - 所有客户端、后台和外部集成都经过 API 契约
   - 使用 OpenAPI 统一接口描述和 SDK 生成

4. 账务绝对优先
   - 钱包、充值、提现、下注、派奖、返利、对账必须优先保证一致性和可追踪性

5. 兼容迁移而非平移复制
   - 兼容老系统数据和业务流程
   - 不照搬旧表结构和存储过程设计

6. 可观测性前置
   - 日志、指标、链路追踪、审计、对账从一开始纳入设计

建议采用“双仓并行”而不是在当前 PHP 仓库上硬叠新架构：

## 6. 领域模块设计

### 6.1 模块划分

| 模块 | 主要职责 | 老系统对应区域 | 新系统关键实体 |
| --- | --- | --- | --- |
| `auth` | 登录、注册、设备会话、验证码、令牌刷新 | `UsersAction`、登录存储过程 | UserCredential、Session、Device |
| `user` | 资料、等级、头像、状态、实名、黑白名单 | `users` 表、资料相关页面 | UserProfile、UserStatus、VipLevel |
| `wallet` | 主余额、保险箱余额、账变流水、冻结、解冻 | `users.points/back`、账变表 | WalletAccount、LedgerEntry、FreezeRecord |
| `payment` | 充值订单、提现申请、回调处理、出款状态 | `pay_online`、`withdrawals`、后台出款 | DepositOrder、WithdrawOrder、PayoutChannel |
| `game-config` | 游戏定义、玩法、赔率、限额、封盘规则 | `game_config`、后台配置页 | GameType、PlayRule、OddsTemplate、LimitPolicy |
| `issue-draw` | 期号生成、开奖采集、封盘、开盘、结果发布 | `caiji/`、开奖过程 | GameIssue、DrawResult、DrawSource |
| `betting` | 投注提交、校验、下注订单、撤单、自动投注 | `GameAction`、下注存储过程 | BetOrder、BetItem、AutoBetRule |
| `settlement` | 派奖、返利、排行榜、红包、补单、补偿 | 活动返利、开奖结算、跑批 | SettlementJob、RewardRule、RewardRecord |
| `campaign` | 活动配置、领取规则、首充/日返/周返 | `ActivityAction` | Campaign、CampaignTask、ClaimRecord |
| `agent` | 邀请、代理树、返佣、团队报表 | 推广、推荐奖励、代理逻辑 | AgentNode、InviteRelation、CommissionRecord |
| `admin` | 后台配置、风控操作、审计查询、报表导出 | `admin/` 全部页面 | AdminUser、Role、Permission、AuditLog |
| `risk-control` | 限频、黑名单、设备/IP 风控、额度管控 | 正则拦截、后台人工控制 | RiskRule、RiskHit、ReviewTask |
| `notification` | 站内信、公告、推送、短信/邮件接口 | 公告、短信脚本 | Notice、PushTask、Template |
| `scheduler` | 定时任务、日结、周结、补偿扫描 | `caiji/` 和脚本任务 | JobDefinition、JobExecution |

### 6.2 重点模块说明

#### `wallet`

这是整个迁移的第一优先级模块，必须承担：

- 金额变更唯一来源
- 账变幂等
- 充值、提现、下注、派奖、返利统一记账
- 审计与对账能力

设计要求：

- 金额统一使用 `long` 存储“千分单位”金额，兼容老系统 `/1000` 规则
- 对外 API 返回金额字符串或 `BigDecimal` 格式，禁止直接暴露内部精度
- 所有资金操作必须写入双向账变摘要字段：`before_amount / delta / after_amount / biz_type / biz_id`

#### `betting`

必须承担：

- 前端请求校验
- 游戏状态校验
- 用户资格校验
- 限额与限频
- 赔率快照固化
- 投注成功后的账务预扣或直接扣减

目标是不再依赖 `web_tz_*` 存储过程作为核心业务主入口。

#### `issue-draw`

必须把当前 `caiji/` 中的开奖采集与期号推进收敛为标准服务：

- 抓取第三方开奖结果
- 标准化解析
- 幂等入库
- 广播开奖事件
- 触发结算
- 记录数据源与延迟

#### `settlement`

必须将当前“开奖后散落在 SQL 过程和跑批中的规则”迁移到应用层：

- 普通派奖
- 自动投注续投
- 亏损返利
- 排行奖
- 红包
- 首充与推荐奖励

## 7. 数据架构规划

### 7.1 数据库重构原则

1. 不照搬旧表族
   - 老系统按游戏复制整套表的方式不再延续

2. 优先建立通用模型
   - `game_type`
   - `game_issue`
   - `bet_order`
   - `bet_order_item`
   - `draw_result`
   - `settlement_record`
   - `wallet_account`
   - `ledger_entry`

3. 少量特化表按玩法扩展
   - 对确有特殊结构的玩法允许建立扩展表
   - 但扩展表只保存玩法特有字段，不复制整套账务和流程表

### 7.2 核心数据对象

| 对象 | 说明 |
| --- | --- |
| `wallet_account` | 用户钱包账户，含主余额、保险箱、冻结金额 |
| `ledger_entry` | 账变流水，所有资金动作必须落这里 |
| `deposit_order` | 充值订单 |
| `withdraw_order` | 提现订单 |
| `game_issue` | 游戏期号和状态 |
| `bet_order` | 一次提交的下注单 |
| `bet_order_item` | 下注子项 |
| `odds_snapshot` | 下单时赔率快照 |
| `draw_result` | 开奖结果 |
| `settlement_record` | 结算结果 |
| `campaign_reward_record` | 活动奖励记录 |
| `agent_commission_record` | 代理返佣记录 |
| `admin_audit_log` | 后台操作审计 |

### 7.3 数据迁移原则

1. 主数据与交易数据分层迁移
   - 用户、配置、游戏定义属于主数据
   - 充值、提现、下注、账变属于交易数据

2. 历史交易分波次迁移
   - 先迁近 `90~180` 天热数据
   - 冷数据归档后按需补迁

3. 存储过程先“对照”，后“退场”
   - 第一阶段允许新服务对照读取老过程结果用于验证
   - 一旦新服务稳定，逐步停止以存储过程作为核心逻辑

4. 迁移全程可追溯
   - 每批迁移必须产出批次号、校验摘要、成功/失败记录和重试日志

### 7.4 Redis 规划

- 用户会话黑名单与 Refresh Token 辅助态
- 热门配置缓存
- 游戏期号与封盘倒计时缓存
- 限频与风控计数器
- 幂等键和短期锁

### 7.5 MQ 事件规划

建议初期统一使用 RabbitMQ，主题可包括：

- `wallet.changed`
- `payment.deposit.completed`
- `payment.withdraw.approved`
- `issue.opened`
- `draw.published`
- `bet.placed`
- `bet.settled`
- `campaign.reward.created`
- `agent.commission.created`

## 8. 客户端规划

### 8.1 Flutter 用户端

用户端统一覆盖：

- 登录注册
- 首页与游戏大厅
- 游戏下注页
- 钱包
- 充值提现
- 活动中心
- 个人中心
- 站内消息

Flutter 端要求：

- 所有业务 API 经 OpenAPI 生成客户端
- 实时信息通过 WebSocket 订阅
- UI 以移动优先，再适配 Web
- 金额、赔率、倒计时、开奖结果等使用统一格式化层

### 8.2 Next.js 后台端

后台管理系统建议独立建设，原因如下：

- 当前后台页面数量多、表格重、运营动作密集
- 需要更适合复杂管理页面的 Web 技术栈
- 后续报表、筛选、导出、批量审批效率更高

后台端范围建议包括：

- 用户查询与封禁
- 充值提现审核
- 游戏配置
- 赔率与限额
- 活动配置
- 风控命中列表
- 报表导出
- 审计日志

## 9. 接口与集成策略

### 9.1 API 设计原则

- 外部接口统一 RESTful + OpenAPI
- 后台管理操作接口与用户端接口分域管理
- 内部模块间优先使用模块内服务调用，跨模块通过领域事件解耦

### 9.2 第三方集成

需要统一抽象以下外部能力：

- 支付渠道
- 短信服务
- 文件存储
- 开奖数据源

每类外部集成都必须具备：

- 统一接口
- 沙箱或模拟器
- 请求响应日志
- 重试与超时策略
- 熔断与降级策略

## 10. 安全、审计与可观测性

### 10.1 安全要求

- Spring Security 统一鉴权
- 用户端采用 `JWT + Refresh Token + 设备会话`
- 后台端必须启用 `RBAC + 审计日志 + 二次确认`
- 敏感配置使用环境变量或密钥管理，不写入仓库
- 所有账务动作必须保留请求来源、操作者、设备/IP、业务单号

### 10.2 审计要求

后台以下操作必须强制审计：

- 提现审核
- 手工加减款
- 封禁/解封
- 活动补发
- 赔率调整
- 游戏开关变更

### 10.3 可观测性要求

- 结构化日志
- Prometheus 指标
- TraceId 全链路透传
- 关键业务仪表盘
- 核心告警：支付失败、提现堆积、开奖延迟、结算异常、账实不符

## 11. 迁移策略

### 11.1 总体迁移方式

采用“绞杀者”模式分阶段替换：

1. 老系统继续承载生产
2. 新系统先从旁路能力建设开始
3. 再逐步接管核心链路
4. 最后完成流量切换和老系统下线

### 11.2 迁移顺序

建议顺序如下：

1. 基础平台与工程骨架
2. 认证、用户、钱包总账
3. 充值与提现
4. 游戏配置、期号、开奖采集
5. 下注与结算
6. 活动、代理、通知
7. 后台运营系统
8. 数据补迁、对账、切换、下线

### 11.3 新老并行原则

- 新服务上线初期以“读校验、影子跑、旁路核对”为主
- 高风险写操作切换前必须至少完成一轮全链路演练
- 账务和结算阶段禁止无校验双写
- 任一阶段都必须保留老系统回退入口

## 12. 分阶段路线图

## 13. 里程碑与验收口径

| 里程碑 | 标志 | 验收口径 |

- 充值/提现关键链路成功率达到目标 SLA
- 开奖延迟与结算延迟满足业务门槛
- 关键 API `p95` 达标
- 审计链路完整

## 15. 风险清单

| 风险 | 说明 | 应对策略 |
| --- | --- | --- |
| 存储过程口径不清 | 老规则分散在 SQL 中 | 逐条盘点并建立对照测试 |
| 账务迁移风险 | 老系统金额单位和流水口径复杂 | 先做账本模型和对账工具 |
| 游戏规则复杂 | 多游戏、多玩法差异大 | 先迁主流玩法，保留特化扩展点 |
| 开奖源不稳定 | 第三方数据可能延迟或异常 | 多源兜底、延迟监控、人工补单 |
| 运营后台重度依赖旧页面 | 切换时阻力大 | 后台单独建设，先补高频场景 |
| 团队节奏失衡 | 客户端、后端、数据迁移依赖强 | 固定里程碑和跨流工作台账 |

## 16. 本规划的关键决策

1. 不在老库上继续加大规模新功能
2. 新系统以模块化单体为第一目标形态
3. 钱包总账先于游戏迁移
4. 后台作为独立 Web 应用建设
5. 金额内部精度沿用千分单位
6. 所有迁移都必须可回滚、可对账、可审计

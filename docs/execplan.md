# dd28 改造迁移执行计划

## 1. 文档定位

本文档用于把 [plans.md](./plans.md) 中的总体规划落到可执行层，覆盖：

- 阶段拆解
- 工作流划分
- 交付物
- 依赖关系
- 验收标准
- 切换与回滚策略

本执行计划默认基于以下前提：

- 主方案为 `Kotlin + Spring Boot + Flutter`
- 后台同步采用 `Next.js`
- 团队规模为 `8~11` 人的最小可交付编制
- 迁移采用“新老并行 + 分波次切换”

---

## 2. 总周期建议

### 2.1 建议总周期

在团队配置基本到位的前提下，建议以 `9~12 个月` 为现实目标。

### 2.2 建议节奏

| 波次 | 周期建议 | 目标 |
| --- | --- | --- |
| Wave 0 | 第 1~4 周 | 立项、摸底、口径冻结 |
| Wave 1 | 第 5~8 周 | 工程骨架、基础设施、CI/CD |
| Wave 2 | 第 9~16 周 | `auth + user + wallet` |
| Wave 3 | 第 17~22 周 | `payment` |
| Wave 4 | 第 23~30 周 | `game-config + issue-draw + realtime` |
| Wave 5 | 第 31~40 周 | `betting + settlement` |
| Wave 6 | 第 41~48 周 | `campaign + agent + admin + cutover` |

说明：

- Flutter、Next.js、测试和运维工作会穿插并行，不是等后端全部做完再开始。
- 若团队不足，优先延长 Wave 4~6，不要压缩 Wave 2~3。

## 3. 工作流拆分

建议按六条主工作流并行推进。

| 工作流 | 负责内容 | 主要负责人 |
| --- | --- | --- |
| `WS1` 产品与口径 | 业务梳理、原型、规则冻结、验收标准 | 产品/业务分析 |
| `WS2` 平台与后端 | Kotlin 服务、领域模型、API、任务、对账 | 后端团队 |
| `WS3` 数据迁移 | 老表分析、迁移脚本、校验、回填、归档 | 数据工程/后端 |
| `WS4` 用户端 | Flutter 三端 UI、API 接入、实时体验 | Flutter 团队 |
| `WS5` 后台运营端 | Next.js 后台、报表、审批、审计 | Web 团队 |
| `WS6` 质量与发布 | 自动化测试、环境、监控、灰度、回滚 | QA + DevOps |

---

## 4. 执行原则

1. 每个阶段必须有“可演示结果”，不能只有代码提交
2. 每个模块切换前必须有对账或结果对照
3. 钱包、支付、结算永远优先于 UI 完整度
4. 先接高频场景，再补长尾功能
5. 切换时优先灰度、影子跑和旁路验证
6. 所有生产切换必须有明确回滚开关

### 阶段 1：新平台基础骨架

### 目标

- 让新平台具备可开发、可部署、可测试、可观测的底座

### 主要任务

#### 后端

- 初始化 `backend` 工程

- 接入 Spring Security、Flyway、Redis、RabbitMQ
- 建立 OpenAPI、统一异常、统一返回体、统一 TraceId

#### Flutter

- 初始化 `user_app`
- 建立主题、路由、环境配置、多端打包流程
- 接入 OpenAPI 生成客户端

#### Next.js

- 初始化 `admin_web`
- 建立登录框架、布局、权限路由、基础表格页

#### 运维

- 建立 dev / sit / uat 环境
- 接入 CI/CD
- 接入日志、指标、告警、链路追踪

### 输出物

- 新仓代码骨架
- 基础环境
- 自动构建与部署流程
- 最小演示页面与健康检查接口

### 退出标准

- 三端均可构建和部署
- 后端可连接 MySQL / Redis / MQ
- 基础监控可用

## 10. 生产切换 Runbook

### 10.1 切换前

- 冻结变更窗口
- 确认所有依赖服务健康
- 执行最终增量迁移
- 确认关键告警面板正常
- 通知业务、客服、运营、技术值守人员

### 10.2 切换中

- 先切只读查询流量
- 再切低风险写流量
- 最后切资金和下注核心写流量
- 实时观察支付、提现、开奖、下注、结算指标

### 10.3 切换后

- 核对关键订单量
- 核对资金流水
- 核对开奖与结算延迟
- 持续观察至少一个完整业务周期

## 11. 回滚策略

以下场景必须触发回滚评估：

- 钱包账务异常
- 充值或提现连续失败
- 开奖延迟超出阈值
- 下注提交异常率过高
- 结算错误或大面积补偿

回滚原则：

- 先关闭新流量入口
- 保留已完成数据日志
- 回切旧系统写流量
- 执行差异核对和后续补偿

## 15. 结论

本执行计划的核心不是“把老 PHP 页面一页页翻译成新技术”，而是按业务价值链重建：

1. 先建底座
2. 再建账务
3. 再接支付
4. 再接开奖
5. 再接下注结算
6. 再迁活动代理后台
7. 最后切流量和下线旧系统

如果后续需要继续细化，我建议下一步直接输出三份补充文档：

- 领域模型与表结构草案
- 存储过程迁移清单
- 第一阶段 `auth + user + wallet` 的 API 与数据库设计稿
